# 第5章 本番環境への道 ― 仕様書

## 章の概要

**章の目的**: サンプルアプリのバンドルサイズ最適化、ブラウザ互換性対応を行い、KubernetesでWASMワークロードをデプロイする方法を習得する。

**目標ページ数**: 10p

## 前提知識

この章を読むために必要な前提:
- 第4章の内容（サンプルアプリのバッチ処理実装済み）
- Kubernetesの基本操作（kubectl、Pod、Deployment）

## 節の構成

### 5.1 バンドルサイズ最適化

**概要**: wasm-optによる最適化、不要コードの除去、圧縮（gzip/brotli）を適用し、WASMバイナリのサイズを削減する。

**図表**:
- 表5.1: 最適化手法別のバイナリサイズ比較

**キーポイント**:
- `wasm-opt -Oz` で大幅にサイズ削減可能
- Cargo.tomlのリリースプロファイル設定も重要

**サンプルアプリ要件**:
- wasm-optを適用してバイナリサイズを100KB以下に削減

### 5.2 ブラウザ互換性

**概要**: 主要ブラウザのWASMサポート状況、フォールバック戦略、Feature Detectionの実装を解説する。

**図表**:
- 表5.2: ブラウザ別WASMサポート状況
- 図5.1: Feature Detectionフロー

**キーポイント**:
- 主要ブラウザは全てWASM対応済み
- `WebAssembly.instantiate` の存在チェックで判定可能

### 5.3 KubernetesでWASMを動かす

**概要**: containerd + runwasiを使ってKubernetes上でWASMワークロードを実行する。サンプルアプリのCLI版（WASI対応）を作成し、Podとしてデプロイする。

**図表**:
- 図5.2: Kubernetes上のWASM実行アーキテクチャ（containerd → runwasi → WASMバイナリ）
- 図5.3: コンテナ vs WASMワークロードの起動時間比較

**キーポイント**:
- runwasiがcontainerdのシムとしてWASMを実行する
- コンテナより起動が高速で、リソース効率が良い

**サンプルアプリ要件**:
- WASI対応のCLI版を `cli/` に作成
- Kubernetesマニフェスト（Deployment、Service）を作成

### 5.4 パフォーマンス分析とまとめ

**概要**: 本書全体の振り返りとして、WASMの現在地と将来展望をまとめる。Component Model、WASI Preview 2等の動向にも触れる。

**図表**:
- 図5.4: WASMエコシステムのロードマップ

**キーポイント**:
- WASMは急速に進化しており、今後も適用範囲が拡大する
- Component Modelにより言語間の相互運用性が向上する見込み
