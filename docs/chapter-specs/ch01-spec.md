# 第1章 WASMの正体 ― 仕様書

## 章の概要

**章の目的**: WebAssemblyの本質を「ポータブルなバイトコード」として理解し、ブラウザからKubernetesまでの活用範囲を把握する。

**目標ページ数**: 8p

## 前提知識

この章を読むために必要な前提:
- いずれかのプログラミング言語での開発経験
- コンテナ（Docker）の基本概念

## 節の構成

### 1.1 WASMとは何か

**概要**: WASMのバイナリフォーマットとしての定義、スタックベースの仮想マシン、モジュール構造を解説する。「言語」ではなく「コンパイルターゲット」であることを強調する。

**図表**:
- 図1.1: WASMの位置づけ（ソースコード→コンパイラ→WASMバイナリ→実行環境）

**キーポイント**:
- WASMは言語ではなくバイナリフォーマットである
- 複数の言語からコンパイル可能

### 1.2 なぜWASMか ― JavaScriptの限界

**概要**: JavaScriptのパフォーマンス上の制約と、WASMが解決する問題を説明する。JITコンパイルとAOTコンパイルの違いにも触れる。

**図表**:
- 図1.2: JavaScript vs WASMの実行モデル比較

**キーポイント**:
- JavaScriptはJIT、WASMはAOTに近い実行モデル
- 予測可能なパフォーマンスがWASMの強み

### 1.3 ブラウザの外へ ― WASIとKubernetes

**概要**: WASIによるOS機能アクセス、Docker/Kubernetes時代におけるWASMの位置づけを解説する。コンテナとWASMの比較を行う。

**図表**:
- 図1.3: コンテナ vs WASMの実行モデル比較
- 表1.1: コンテナとWASMの特性比較

**キーポイント**:
- WASIによりWASMはブラウザ外でも動作する
- コンテナより軽量・高速な起動が可能

### 1.4 本書で作るもの

**概要**: サンプルアプリ「WASM Image Filter」の完成イメージを提示し、各章での構築ステップを概観する。

**図表**:
- 図1.4: サンプルアプリの完成イメージと各章の対応

**キーポイント**:
- 画像フィルタアプリを段階的に構築する
- 最終的にKubernetesにデプロイする
