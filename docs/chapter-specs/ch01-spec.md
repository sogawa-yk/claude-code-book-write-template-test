# 第1章 WASMとは何か ― バイナリフォーマットの本質 ― 仕様書

## 章の概要

**章の目的**: WebAssemblyが「ブラウザ用の技術」ではなく「ポータブルなバイトコード」であることを理解し、なぜ今WASMが重要なのかを把握する。

**目標ページ数**: 8p

**所属する部**: 第I部 WASMの正体

この章を読むために必要な前提:

- Webブラウザの基本的な動作原理

## 節の構成

### 1.1 WebAssemblyの正体 ― バイナリフォーマットとしてのWASM

**概要**: WASMは言語ではなくバイナリフォーマットであることを明確にする。コンパイルターゲットとしてのWASMの位置づけを、JavaバイトコードやLLVM IRと比較しながら解説する。

**図表**:
- 図1.1: WASMの位置づけ ― ソースコードからバイナリまでの変換フロー

**キーポイント**:
- WASMは「言語」ではなく「コンパイルターゲット」である
- 複数の言語（Rust、Go、C/C++等）からコンパイル可能

### 1.2 なぜWASMか ― JavaScriptの限界とWASMの強み

**概要**: JavaScriptの性能的限界を示し、WASMがどのような場面で優位性を持つかを解説する。サンドボックスモデルによるセキュリティ特性にも触れる。

**図表**:
- 図1.2: JavaScript vs WASM ― 実行モデルの比較
- 表1.1: JavaScriptとWASMの特性比較

**キーポイント**:
- WASMはネイティブに近い速度で動作する
- サンドボックス実行によるセキュリティの保証
- JavaScriptの代替ではなく補完である

### 1.3 ブラウザを超えて ― WASIとサーバーサイドWASM

**概要**: WASIの登場によりWASMがブラウザ外で実行可能になった経緯を解説する。Docker/Kubernetes時代におけるWASMの可能性を示し、本書の全体像を提示する。

**図表**:
- 図1.3: WASMの実行環境 ― ブラウザからサーバーサイドまで

**キーポイント**:
- WASIにより、ファイルI/Oやネットワークアクセスが可能に
- 「Write once, run anywhere」の新しい形
- コンテナ技術との関係性
